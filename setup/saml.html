SAML Service Provider Setup

OPTiM Store API Docs

SAML Protocol に従い ID 連携を行うには、御社サービスは OPTiM Store を ID Provider とした SAML Service Provider (SP) になる必要があります。

そのため、まずテナント契約直後に 御社サービスから OPTiM Store に対して SAML Service Provider を登録する必要があります。

OPTiM Store では、OpenID Connect Client Setup と同じような方法で、SAML SP 登録を受け付けています。

事前設定項目

以降、本ドキュメントでは、OPTiM Store の SAML Service Provider Registration API Endpoint の URL を以下の値と仮定して説明を進めます。

  • https://optim-store.optim.co.jp/saml2/service_providers

また Tenant Contract Setup 時に御社サービス側で生成した RSA 秘密鍵を利用してリクエストに署名をする必要があるので、事前にテナント契約が成立している必要があります。

SAML Service Provider Registration Request

テナント契約が成立した後に、御社サービスから OPTiM Store の SAML Service Provider Registration API Endpoint に以下のような JSON データを送ります。(実際には以下の JSON に JWS 仕様にしたがって署名したデータを送ります)

{
  "iss": "https://your-service.example.com/contract/api",
  "aud": "https://optim-store.optim.co.jp/saml2/service_providers",
  "entity_id": "https://your-service.example.com/saml2",
  "nameid_format": "persistent",
  "callback_url": "https://your-service.example.com/saml2/callback",
  "iat": 1464053706,
  "exp": 1464054306
}

リクエストJSONの各要素について

key description
iss リクエスト発行者 (issuer) の識別子です。御社サービスの Contract Start API Endpoint URL となります。
aud リクエスト受信者 (audience) の識別子です。OPTiM Store の SAML Service Provider Registration API Endpoint となります。
entity_id SP Entity ID です。省略すると後述の callback_url を entity_id として扱います。
nameid_format persistent か email を指定可能です。ただし Email をユーザー識別子として用いるとセキュリティ上の問題を引き起こしやすくなるため、persistent の利用を強く推奨します。
callback_url OPTiM Store で認証されたユーザーが御社サービスにリダイレクトされて返される際の Callback Endpoint URL です。
iat リクエスト生成日時 (issued_at) を示す Unix Timestamp です。
exp リクエストの有効期限 (expires_at) を示す Unix Timestamp です。

署名付きリクエスト

実際の POST Body は、上記の JSON に御社サービス側の当該テナント契約に紐付いた RSA 秘密鍵で署名したものを software_statement という key に wrap した JSON データになります。

その時、署名に使用した鍵の ID (kid) を JWS Header の kid に指定してください。OPTiM Store はその kid をもとに対応するテナント契約を特定します。

署名付き Software Statement の生成方法については OpenID Connect Client Setup の「署名付きリクエスト」の章を参照してください。

SAML Service Provider Registration Response

OPTiM Store 側でのリクエスト検証が成功すると、以下のような JSON Response が返されます。

{
  "entity_id": "https://federation.optim.co.jp/saml2/o4K32SHuf0uQ1lbOD7GglC3GSaOrfDuzqASwHU0OQBs/assertion",
  "single_sign_of_service": {
    "binding": "http_redirect",
    "location": "https://federation.optim.co.jp/saml2/o4K32SHuf0uQ1lbOD7GglC3GSaOrfDuzqASwHU0OQBs/assertion"
  },
  "nameid_format": "persistent",
  "certificate": "-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----\n",
  "idp_metadata": "\u003c?xml version=\"1.0\" encoding=\"UTF-8\"?\u003e\n\u003cmd:EntityDescriptor xmlns:md=\"urn:oasis:names:tc:SAML:2.0:metadata\"..."
}

idp_metadata には SAML IdP Metadata XML が含まれます。

その他の属性値は idp_metadata に含まれる値と同じなので、無視していただいても構いませんし、IdP Metadata XML を Parse する代わりに JSON から必要なパラメータを取得していただいても構いません。

このレスポンスを元に、御社サービス側テナントを OPTiM Store SAML IdP テナントと関連付けてください。
