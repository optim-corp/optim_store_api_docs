OpenID Connect Federation

OPTiM Store API Docs

OpenID Connect Protocol に従い ID 連携を行うには、御社サービスは OPTiM Store を ID Provider とした OpenID Connect Relying Party になる必要があります。

エンドユーザー様が御社サービスにログインするたびに、御社サービスは OPTiM Store に OpenID Connect プロトコルに従った認証リクエストを送信し、返されるレスポンスをもとに、エンドユーザー様を御社サービスにログインさせることになります。

要素技術

OpenID Connect は、OAuth 2.0 をベースとした ID 連携プロトコルであり、Google Sign-in や Yahoo! ID 連携など、コンシューマ向けサービスで広く用いられています。

また OAuth 2.0 をベースとしたプロトコルのため、Facebook ログインともよく似たフローになります。

OpenID Connect や OAuth 2.0 の仕様は、以下のサイトで翻訳版をご覧いただけます。
実装時にはこれらの情報も参考にしてください。

  • http://openid-foundation-japan.github.io

OAuth 2.0 についてはこちら。

  • http://openid-foundation-japan.github.io/rfc6749.ja.html
  • http://openid-foundation-japan.github.io/draft-ietf-oauth-v2-bearer-draft11.ja.html

OpenID Connect についてはこちら。

  • http://openid-foundation-japan.github.io/openid-connect-basic-1_0.ja.html

以下のサイトでは OpenID Connect やその関連仕様に準拠した各言語でのライブラリが紹介されています。

  • http://openid.net/developers/libraries/

事前設定項目

OpenID Connect を利用して OPTiM Store と御社サービスとの ID 連携を実現するには、御社サービスの各テナントを OPTiM Store に OpenID Connect Client として登録し、Client ID & Secret を取得する必要があります。

また OPTiM Store に御社サービス側の OpenID Connect Callback URL (redirect_uri, HTTPS 必須) を登録する必要もあります。OpenID Connect はリダイレクトを介した Token のやり取りを行うことになりますが、この Callback URL は OPTiM Store から御社サービス側にユーザー様を返す時に利用されます。

これらは OpenID Connect Client Setup にしたがって自動設定を行うこともできますが、自動設定を行わない場合は書面ベースやメールベースで行う必要があります。(自動設定を行わない場合のフローについては現在詳細検討中です)

御社サービスから OPTiM Store へ認可リクエスト送信

ユーザー様をログインさせるタイミングで、御社サービスから OPTiM Store の認可エンドポイント (Authorization Endpoint) へユーザー様をリダイレクトさせます。

以降、本ドキュメントでは、OPTiM Store の Authorization Endpoint URL を以下の値と仮定して説明を進めます。

  • https://federation.optim.co.jp/connect/authorization

認証リクエストには、以下のパラメータを含めます。

Key Value
client_id 事前登録時に取得した Client ID
scope openid 固定
response_type code 固定
redirect_uri 事前登録した御社サービス側の OpenID Connect Callback URL
state 御社サービス側のセッションに紐付いた任意の文字列。redirect_uri への Callback 時にそのままの値が返されるので、CSRF 対策に利用してください。
prompt OPTiM Store 側でユーザー様にパスワード入力を強制 (強制ログイン) する場合には login という値を指定してください。それ以外の場合にはこのパラメータは不要です。

以下に Ruby での Authorization Request 送信のコンセプトコードを記載します。(実際の利用時には OpenID Connect ライブラリ等の利用を推奨します)

params = {
  client_id: '<your-client-id-for-the-tenant>',
  scope: 'openid',
  response_type: 'code',
  redirect_uri: 'https://your-service.example.com/optim-store/callback',
  state: '<session-binded-opaque-value>'
}
authorization_endpoint = 'https://federation.optim.co.jp/connect/authorization'

authorization_endpoint_with_request_params = [authorization_endpoint, params.to_query].join('?')
# https://federation.optim.co.jp/connect/authorization?
#   client_id=%3Cyour-client-id-for-the-tenant%3E&
#   redirect_uri=https%3A%2F%2Fyour-service.example.com%2Foptim-store%2Fcallback&
#   response_type=code&scope=openid

redirect_to authorization_endpoint_with_request_params

これにより、ユーザー様は、自身の端末上のブラウザで、OPTiM Store のログインページにアクセスすることになります。

OPTiM Store から御社サービスへの Callback

OPTiM Store は御社サービスからの認可リクエストを受け取ると、ユーザー様を認証したのち、御社サービスの redirect_uri にユーザー様をリダイレクトして戻します。

この Callback には、「認可コード (Authorization Code, code)」と呼ばれる Onetime Token と、認可リクエスト時に御社サービス側で指定した state 値が Query に付与されています。
code は次のセッションで認証結果を取得する際に必要になります。
なお、state がリクエスト送信時のブラウザセッションと紐付いていない場合は CSRF 攻撃の可能性があるため、その Callback リクエストはエラーとして受け付けないようにしてください。

以下に Callback の例を記載します。

https://your-service.example.com/optim-store/callback?code=<onetime-token>&state=<exact-value-you-sent>

code が付与されていない場合は、ユーザーのログイン失敗等によるエラーレスポンスです。その場合は error というパラメータが付与されています。

以下にエラー時のリダイレクト URL の例を記載します。

https://your-service.example.com/optim-store/callback?error=access_denied&state=<exact-value-you-sent>

OPTiM Store から認証結果を取得

Callback 時に受け取った Authorization Code を OPTiM Store の Token Endpoint に送信 (Token Request) することで、認証結果を示すトークン (ID Token) を受け取ることができます。

以降、本ドキュメントでは、OPTiM Store の Token Endpoint URL を以下の値と仮定して説明を進めます。

  • https://federation.optim.co.jp/connect/token

ID Token を取得するためには、上記 Token Endpoint に以下のパラメータを application/ x-www-form-urlencoded 形式で POST します。

key description
client_id 事前登録時に取得した Client ID
client_secret 事前登録時に取得した Client Secret
grant_type authorization_code 固定
code 取得した code 値
redirect_uri 認証リクエスト時に指定した redirect_uri

(client_id & client_secret は Basic Authentication Header に username & password として指定することもできます)

以下に Ruby での Token Request 送信のコンセプトコードを記載します。(実際の利用時には OpenID Connect ライブラリ等の利用を推奨します)

params = {
  client_id: '<your-client-id-for-the-tenant>',
  client_secret: '<your-client-secret-for-the-tenant>',
  grant_type: 'authorization_code',
  code: '<onetime-token>',
  redirect_uri: 'https://your-service.example.com/optim-store/callback'
}
token_endpoint = 'https://federation.optim.co.jp/connect/token'

response = HTTPClient.post(token_endpoint, params)
response.body
# {
#   "access_token": "fake",
#   "token_type": "bearer",
#   "expires_in": 300,
#   "id_token": "<id-token>"
# }

リクエスト成功時には以下のような属性値が JSON 形式で返されます。

key description
access_token 一部のライブラリでエラーになることを防ぐ目的で OAuth 2.0 に準拠するためだけに発行される Token です。OPTiM Store との連携ではこの Token は利用しません。
token_type bearer 固定
expires_in access_token の有効期間 (秒数) です。access_token は利用しないので無視してください。
id_token ID Token です。後述の手順でこちらの Token をパースして認証結果を検証することになります。

リクエスト失敗時にはエラー内容が JSON で返されます。40x 系のエラーの場合は、同じ Authorization Code を使って Token Request をリトライしても同じくエラーになりますので、認可リクエストからやり直すようにしてください。

ID Token の検証

OpenID Connect の ID Token は、JSON Web Token (JWT) という仕様にしたがって OPTiM Store によって署名が付与された JSON データです。

JWT の仕様はこのドキュメントの冒頭で紹介したサイトで翻訳版およびライブラリリストが公開されています。

また以下のサイトでも、各言語ごとの JWT ライブラリが紹介されているので、参考にしてください。

  • https://jwt.io

以下に Ruby による ID Token デコード用のコンセプトコードを記載します。

require 'json-jwt'

id_token = JSON::JWT.decode '<id-token>', :skip_verification
# {
#   "iss": "https://federation.optim.co.jp",
#   "aud": "<your-client-id-for-the-tenant>",
#   "sub": "<optim-store-user-guid>",
#   "auth_time": 1464320207,
#   "iat": 1464320208,
#   "exp": 1464320508
# }

ID Token の中には以下のような属性値が入っています。

key description
iss Token 発行者 (issuer) の識別子です。https://federation.optim.co.jp 固定です。
aud Token 受信者 (audience) の Client ID です。
sub 認証されたユーザー様 (subject) の OPTiM Store 上の GUID です。
auth_time OPTiM Store 上でのユーザー様の最終ログイン日時 (authenticated_at) を示す Unix Timestamp です。強制認証リクエストを送信した場合には、リクエストパラメータの改ざんによる強制認証回避を防ぐため、この値が直近の日時であることを確認してください。
iat ID Token の生成日時 (issued_at) を示す Unix Timestamp です。
exp ID Token の有効期限 (expires_at) を示す Unix Timestamp です。

ID Token に含まれる上記の属性値を以下の通り検証してください。

  • iss が https://federation.optim.co.jp と一致すること。
  • aud が該当テナントに発行された Client ID と一致すること。
  • exp が未来の日時であること。
  • iat が現在もしくは過去の日時であること。
  • 強制認証要求時には auth_time が直近の日時 (5秒以内等) であること。

なお、OPTiM Store Token Endpoint へのリクエスト時に TLS サーバ認証を行うことで、そのレスポンスは OPTiM Store サーバーから返却されたことが保障されるため、ID Token 自体の署名検証は省略可能です。

ID Token 検証が成功したら、該当テナント内に存在する sub に紐付いたユーザー様を認証済としてください。

sub 値に紐付いたユーザー様は、事前にプロビジョニング (SCIM) API 経由で登録されているはずです。

もし未知の sub 値を受け取った場合はプロビジョニングが失敗している可能性があるため、一旦リクエストをエラーにした上で当該ユーザー様のライセンス所有の有無を OPTiM Store 側にお問い合わせください。
